# Pontis 本科毕设论文计划

本文围绕 Pontis 的系统价值展开，而不是把论文写成单纯的 BIRD leaderboard 对比。核心问题是：在不训练专用 Text-to-SQL 模型的前提下，一个面向真实数据项目的 agent workspace，能否通过结构化元数据、图谱实体、工具探索和可复用经验，提升通用 LLM 在数据库问答任务中的可靠性、可解释性和可维护性。

## 核心论点

论文主张：

1. Pontis 不是专用 SQL 模型，而是一个面向异构数据源的 data agent 工作空间。
2. Pontis 将数据库 schema、列语义、值分布、关系、消歧和跨库经验组织成可查询图谱实体。
3. 相比 Bash Agent，Pontis 用可复用结构化知识降低每题临时探索成本，并提升 Text-to-SQL 准确率。
4. 相比 CHESS、Alpha-SQL、DeepEye-SQL 等 test-time pipeline，Pontis 的重点不是堆叠候选搜索，而是提供可解释、可缓存、可跨数据源复用的数据工作空间。
5. Pontis 的代价是预处理和 agent 工具调用成本；这些成本需要在论文里单独计量，而不是混在准确率表里。

## 实验对象

主实验保留五个方法：

| 方法 | 本地目录 | 定位 |
|---|---|---|
| Bash Agent | `Bash-Agent` | 无图谱、无专用工具的通用 shell/sqlite agent baseline |
| Pontis | `Pontis` | 本文方法 |
| CHESS | `CHESS` | IR + schema selector + candidate generation + unit test pipeline |
| Alpha-SQL | `Alpha-SQL` | MCTS / test-time search baseline |
| DeepEye-SQL | `DeepEye-SQL` | value retrieval + schema linking + generation/revision/selection pipeline |

主实验不再扩展 RSL-SQL、OpenSearch-SQL、DIN-SQL、DAIL-SQL、MAC-SQL 等方法，避免复现范围失控。这些方法放在相关工作或附录中说明。

## 实验设置

统一约束：

- 使用同一份 BIRD dev 或明确声明的 dev 子集。
- 所有 baseline 使用根目录统一配置中的 DeepSeek LLM 和统一 embedding 模型。
- 固定 temperature、top-p、最大输出长度、SQL 执行超时。
- 固定是否允许执行中间 SQL；如果某方法天然使用执行验证，论文中明确说明这是方法的一部分。
- 不使用 dev gold SQL 更新 prompt、知识库或调参。
- train example 只能作为跨库经验，不允许泄露 dev 答案。
- 每个方法使用自己的 `workspace/baselines/<method>/data` 工作副本，输出、日志和评测结果都写入自己的 workspace。

建议先用 BIRD dev 的 3 个代表数据库做小规模完整实验，再扩展到 full dev：

| 数据库 | 选择原因 |
|---|---|
| `california_schools` | 列名长、公式多、BIRD 风格明显 |
| `card_games` | schema/value 歧义明显，适合错误归因 |
| `codebase_community` | 多表关系和 join 推理较多 |

## 消融实验

目标：证明 Pontis 内部模块不是堆功能，而是对应具体错误类型。

| 设置 | 去掉的能力 | 支撑的论文 claim |
|---|---|---|
| Pontis-full | 无 | 完整系统能力 |
| no-bird-global | 不使用 BIRD train 经验库 | 跨库经验主要迁移 SQL 写作风格 |
| no-disambig | 去掉消歧实体 | 图谱消歧减少 `name` / `type` / `id` 类混淆 |
| no-query | 禁用 SQL 验证查询 | 执行反馈对真实数据库很关键 |
| no-meta-detail | 去掉 AI 生成表/列 detail | 语义化 schema 描述帮助复杂列名理解 |

最低完成：`Pontis-full`、`no-bird-global`、`no-disambig`、`no-query`。

## 错误归因实验

目标：解释为什么 Pontis 仍会错，以及哪些错误可以被系统设计继续改善。

建议从错题中抽样 `N=30` 到 `N=50`，人工归因：

| 错误类型 | 说明 | 是否可由 Pontis 改善 |
|---|---|---|
| BIRD SQL 风格偏好 | percentage 公式、COUNT DISTINCT、输出列契约等 | 可通过 global knowledge 部分改善 |
| 局部 schema 理解错误 | 表/列语义混淆、桥表误用、同名列误判 | 可通过 detail/disambig/关系提取改善 |
| value grounding 错误 | 枚举值、文本格式、代码映射错误 | 可通过 sample/topk/query 改善 |
| benchmark/gold 歧义 | evidence 与 gold 不一致，或题意本身模糊 | 较难改善 |
| 工具行为问题 | 路径、查询、过度探索、输出格式错误 | 可工程修复 |

每类错误至少给 2 个具体 case，包含 question、evidence、predicted SQL、gold SQL、错误原因，以及 Pontis 日志中支持或误导判断的工具信息。

## BIRD Global 迁移实验

目标：证明 BIRD train 的作用不是直接复用 SQL，而是迁移 SQL 写作风格。

| 问题 | 测法 |
|---|---|
| train 中是否存在相关经验 | 对错题用 `search(ref="bird::*:example")` 检索 top-k |
| 检索是否命中同类 SQL 模式 | 人工标注 top-5 是否相关 |
| agent 是否采用经验 | 检查日志是否读取并迁移 |
| 哪些模式可迁移 | percentage、COUNT、CASE WHEN、输出格式 |
| 哪些模式不可迁移 | 当前库特有 schema、局部列名歧义、gold 怪口径 |

这一节可以写成论文的重要发现：

> BIRD train 对 dev 的帮助主要体现在 SQL 写作风格迁移，而不是 schema 或 SQL 片段复用。抽象知识节点比 raw example 检索更稳定。

## 性能实验

当前阶段先不测试数据库变动后的增量更新成本。性能指标需要精简，围绕“建库成本”和“解题成本”两段报告。

| 指标 | 含义 | 主表优先级 |
|---|---|---|
| Preprocess Time | 从空 workspace 生成索引、快照、图谱或任务缓存的 wall-clock 时间 | 必报 |
| Preprocess Size | 预处理产物大小，例如图谱、向量库、snapshot、pickle、日志大小 | 必报 |
| Online Time/Q | 已有必要预处理资产后，每题端到端 wall-clock 时间，报告平均值和 P95 | 必报 |
| LLM Rounds/Q | agent-style 方法中串行决策轮数；最能解释用户等待时间 | 必报 |
| LLM Calls/Q | 每题 LLM 请求数；pipeline/search 方法中比 tool calls 更公平 | 必报 |
| Input Tokens/Q | 每题输入 token；反映 schema、工具结果和历史 trace 的上下文膨胀 | 必报 |
| Output Tokens/Q | 每题输出 token；通常不是主瓶颈，但可解释中间推理/候选生成成本 | 可选 |
| Tool Calls/Q | agent-style 方法中的探索操作数量 | 必报 |
| SQL Exec/Q | 每题执行数据库查询次数，反映执行验证成本 | 必报 |

即席查询性能的核心瓶颈不是最终 SQL 输出本身，而是模型和工具之间的在线交互链路。可以用下面的近似关系解释：

```text
Online Time
≈ Σ LLM latency per round
+ Σ tool execution latency
+ context/token processing cost
+ queue/network overhead
```

其中 `LLM round` 和 `tool call` 不能混为一谈。一轮 LLM 决策可能调用 0 个、1 个或多个工具：

```text
Round 1: LLM -> schema_search, sample_rows, run_sql
Round 2: LLM reads three tool results -> final SQL

LLM Rounds = 2
Tool Calls = 3
```

因此，工具调用次数通常和性能负相关，但它不是跨方法最公平的单一变量。对 Bash Agent 和 Pontis，`LLM Rounds/Q + Tool Calls/Q + Input Tokens/Q` 最能解释 agent 探索成本；对 CHESS、Alpha-SQL、DeepEye-SQL，`LLM Calls/Q + Candidate/Stage Count + Input Tokens/Q` 更能解释 pipeline/search 成本。

| Bash Agent | TBD | TBD | TBD | TBD | TBD | TBD | TBD | TBD | TBD |
| Pontis | TBD | TBD | TBD | TBD | TBD | TBD | TBD | TBD | TBD |

```text
预处理成本：Bash Agent << Pontis ~= Alpha-SQL < CHESS < DeepEye-SQL
在线成本：Bash Agent ~= Pontis < CHESS < DeepEye-SQL <= Alpha-SQL
```

## 公平性能测试协议

性能评估不能只看命令行外层 `time`，因为不同框架的并行度、缓存、断点续跑和内部阶段划分不同。采用“两层计时”：

1. 外层统一计时器：用 `tools/run_with_log.py` 包裹每个命令，记录 wall-clock、退出码、开始/结束时间、命令、工作目录、环境变量、cache 状态。
2. 内部日志解析：从各框架自己的产物中提取每题耗时、每阶段耗时、LLM round/call、token、tool call、SQL 执行次数。

主性能对比只报告：

| 阶段 | 定义 |
|---|---|
| `preprocess_cold` | 删除该方法 workspace 下的索引、快照、图谱和预处理产物后，从零预处理 |
| `online_cold` | 保留必要预处理资产，删除每题运行结果，重新解题 |

`warm` 或 `resume` 可以作为系统分析附表，不放进主性能表。

公平控制项：

| 控制项 | 做法 |
|---|---|
| 数据 | 同一批 `question_id` 和 `db_id`；每个方法自己的数据副本来自同一 canonical copy |
| 并行度 | 主实验设为单 worker；若方法必须内部并行，记录实际 worker 数并不和单 worker 表混用 |
| CPU | 设置 `OMP_NUM_THREADS=1`、`MKL_NUM_THREADS=1`、`OPENBLAS_NUM_THREADS=1`、`TOKENIZERS_PARALLELISM=false`；必要时用 `taskset` 固定 CPU |
| GPU | 主公平实验禁用本地 GPU，设置 `CUDA_VISIBLE_DEVICES=""`；DeepEye 的 `embedding_device` 和 `local_index_device` 固定为 `cpu` |
| 网络 | 不建议人为限速；所有方法顺序运行、避免并发 API 请求、记录 endpoint 和时间段。网络抖动用重复运行和 token/call 指标解释 |
| 缓存 | cold run 前只删除对应方法 workspace 产物，不删除 `.venv` 或下载依赖 |
| 模型 | 根目录 `.env` 和 `experiment_config.yaml` 统一 LLM、embedding、采样参数 |
| SQL timeout | 所有评测统一 SQL 执行超时 |
| 评测 | 用根目录统一 `tools/evaluate_bird_predictions.py` 输出 EX、每题结果和 summary |

结论：CPU/GPU 需要限制，尤其是并行度和 GPU 自动选择；网络不适合做硬限速，但必须顺序运行、记录 API 条件，并补充 LLM call/token 这类不受瞬时网络影响的工作量指标。

## 各方法资源使用

从当前代码库看，各方法主要资源如下：

| 方法 | CPU/并行 | GPU | 网络/API | 本地服务/磁盘 | 公平配置建议 |
|---|---|---|---|---|---|
| Bash Agent | 单进程为主，执行 shell/sqlite | 不需要 | LLM API | 读 SQLite，写运行结果 | 保持单 worker；重点统计 LLM rounds、tool calls、tool result tokens |
| Pontis | `run_bird_benchmark.py` 支持 db/query worker 并行 | 不需要 | LLM + embedding API | Neo4j 本地进程、图谱/日志/结果 | `--db-workers 1 --workers 1`；重点统计 LLM rounds、tool calls、Neo4j/tool latency |
| CHESS | preprocess 固定 `NUM_WORKERS=1`，runner 支持 `--num_workers` | 当前统一远程 embedding 下不需要 | LLM + embedding API | LSH/vector DB/中间 JSON | `--num_workers 1` |
| Alpha-SQL | preprocessor、MCTS、SQL selection 都支持多进程 | 不需要 | LLM + embedding API | LSH/task pickle、MCTS pkl、结果 JSON | `n_processes=1`，selection/preprocess process 数设为 1 |
| DeepEye-SQL | 多阶段 ThreadPool，默认并行度较高 | 默认 `auto` 可能使用 CUDA | LLM + embedding API | vector DB、dataset/stage snapshot | 所有 `n_parallel`/`n_internal_parallel` 设为 1，设备设为 CPU |

如果后面要比较“各方法官方最佳吞吐”，可以单独增加 optimized setting 表；但主论文公平表应使用单 worker/CPU 配置，否则性能差异会混入并行资源差异。

## 输出与日志规范

每个 baseline 的产物固定写到：

```text
workspace/baselines/<method>/
  data/
  preprocess_logs/
  runtime_logs/
  results/
  evaluation/
  manifest.json
```

论文实验记录至少保留：

- `preprocess_logs/`：预处理 stdout/stderr、外层计时 JSON。
- `runtime_logs/`：在线解题 stdout/stderr、外层计时 JSON。
- `results/`：预测 SQL、每题运行信息、方法原生产物。
- `evaluation/`：统一评测结果、summary、错误样本。

## 增量更新实验

暂不放入当前可执行实验。它仍然是 Pontis 的重要系统价值，但现阶段外部 baseline 的增量更新接口不完整，强行测试会变成“重建整个 pipeline”的工程对比，不够公平。

论文处理方式：

- 主实验不报告增量更新成本。
- 系统设计章讨论 Pontis 为什么天然支持实体级增量刷新。
- 总结与展望里提出未来实验：新增列、修改值分布、新增关系、删除表/列，对比 Pontis 局部更新和 pipeline rebuild。

## 当前执行顺序

建议按以下顺序推进：

1. 固定 BIRD 子集、模型、统一评测脚本和 workspace 输出目录。
2. 小子集 smoke test：每个方法跑同一批 `question_id`，确认能产出预测 SQL。
3. 预处理冷启动：清空各自 workspace 产物，记录 `preprocess_cold`。
4. 在线冷启动：保留必要预处理资产，删除每题结果，记录 `online_cold`。
5. 统一评测：所有结果转成统一格式，跑 `tools/evaluate_bird_predictions.py`。
6. 汇总主表：准确率、预处理时间/大小、在线平均/P95、LLM rounds/calls、input tokens、tool calls、SQL exec。
7. 跑 Bash Agent 与 Pontis 的重点对比，先证明核心增益。
8. 复现 CHESS / Alpha-SQL / DeepEye-SQL 中至少 1-2 个强 baseline。
9. 完成 Pontis 消融。
10. 做错题归因和 BIRD global 迁移分析。
11. 写实验章，再回头整理系统设计章。
