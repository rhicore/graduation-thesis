# 如何解决论文缺乏科研性而更像是一个技术报告的问题

- 引入消融实验（Ablation Study）以确立因果联系
    - 护栏消融：对比包含与不包含“桥接表检查”或“SQL消歧检查”时，系统在复杂多表关联题目上的表现差异。
    - 图谱消融：如果去除“基于漏斗过滤模型的隐式关联检测”，仅依赖数据库原生的显式外键，准确率会发生何种变化？
    - 通过量化差异，从科学角度论证这些复杂工程设计的不可替代性。
- 建立严谨的基线对比
    - 引入对比实验。证明 Pontis 系统在应对大规模异构数据时，相较于纯 Prompt 工程方法或现有的检索增强方法，具备统计学意义上的显著优势。若未达到最优性能，则需通过实验数据客观论证该架构在计算效率（Token消耗、工具调用耗时）上的权衡优势。
- 将“文档式罗列”转化为“问题驱动的论证”
    - 当前缺陷：第3章和第4章的撰写方式类似于API文档。例如“智能体配备了14种专用工具，按功能分为以下五类...”，这种叙述缺乏学术深度。
    - 修正策略：将模块设计与 Text-to-SQL 的核心挑战建立强逻辑映射。
    - 不要写“我们开发了工具 A”，而应表述为：“为解决 LLM 在处理领域术语时的幻觉问题，我们引入了机制 A，该机制的理论基础是...”。
    将架构图中的组件视为验证科学假设的“控制变量”，而非单纯的功能堆砌。
- 深化错误分析的理论维度
    - 当前缺陷：表2的错误分类较好，但结论停留在表层（“这说明在复杂查询中，LLM对多表关联路径和条件约束的理解仍有不足”），未能反哺架构设计。
    - 修正策略：针对“WHERE条件偏差”和“JOIN路径偏差”这两类主要错误，探讨当前 Agent 架构的根本性局限。例如，当工具调用次数超过某个特定阈值时，LLM 的上下文注意力是否会发生崩溃？这种失效率与数据库模式图的直径或度数分布是否存在相关性？能够清晰界定系统边界和失效模式，是科学严谨性的重要体现。
