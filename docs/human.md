增量更新问题


---


目前实验的关键问题并不只是继续提升执行准确率，而是需要先明确 Text-to-SQL 任务本身的边界。传统上，Text-to-SQL 可以被抽象为学习一个从自然语言问题和数据库到
  
SQL 的映射函数：

f(query, database) -> sql

但这个建模本身隐含了一个很强的假设：给定同一个 query 和同一个 database，存在唯一正确的 SQL。实际情况往往并非如此。更准确地说，Text-to-SQL 不是一个严格的一
对一函数映射，而更接近从自然语言问题到一组可能 SQL 实现的多值映射。造成这种非唯一性的原因主要有两类。

第一，同一业务目标在 SQL 层面具有多重可达性。也就是说，为了回答同一个业务问题，可能存在多种执行结果相同、语义相近，甚至对真实用户更有用的 SQL 写法。BIRD 数
据集只选择其中一种作为 golden SQL，但这并不意味着该写法是唯一正确的。模型如果生成了另一种业务上合理的 SQL，仍可能因为输出列、排序、格式、数值类型或聚合粒度
与 golden SQL 不一致而被判错。

例如，在 card_games/q390 中，问题询问 ID 1-20 的卡牌颜色和格式。BashAgent 返回了 id, colors, format，其中 id 有助于识别每张卡牌；但 golden SQL 只返回
colors, format，因此额外输出 id 反而导致评测错误。类似地，在 superhero/q763 中，问题要求 Abomination 的 attribute value，BashAgent 返回了 attribute_name,
attribute_value，这对人类阅读更完整，但 golden SQL 只接受 attribute_value。在 toxicology/q226 中，问题明确要求五位小数百分比，BashAgent 使用
printf('%.5f', ...) 输出固定五位小数字符串，而 golden SQL 使用 ROUND(..., 5) 输出数值；两者都符合“保留五位小数”的直觉，但执行结果类型不同会导致判错。这些例
子说明，benchmark accuracy 有时测量的是“是否匹配 golden SQL 的具体输出契约”，而不完全等价于“是否回答了用户问题”。

第二，query 本身往往具有模糊性。自然语言问题通常是用户基于某种数据库先验写出的简短表达，但这些先验并不会完整出现在 query 中。当数据库标注不够丰富、evidence
不够明确，或者同一术语可以映射到多个表列时，agent 只能根据问题文本和数据库内容选择一个最可能的解释。这个解释在业务上可能合理，却未必与 golden SQL 的标注选
择一致。

例如，在 codebase_community/q689 中，问题要求找出“last to edit the post with ID 183”的用户。直觉上可以使用 postHistory 中最后一条编辑记录，也可以使用
posts.LastEditorUserId 字段；但 golden SQL 实际连接的是 posts.OwnerUserId，这与“last to edit”的自然语言表达并不完全一致。不同方法在这个题上选择了不同路径，
说明问题本身存在解释空间。在 student_club/q1376 中，问题问 closed events 中 spend-to-budget ratio 最高的 event。BashAgent 将每个 event 的预算行聚合后计算
SUM(spent)/SUM(amount)，这是一个业务上很自然的“事件级比例”；但 golden SQL 按单条 budget row 的 spent/amount 排序，没有做 event-level aggregation。再如
thrombosis_prediction/q1187，题面说 examined between 两个日期，同时条件来自 GPT 和 ALB。模型可能把 “examined date” 映射到 Examination.Examination Date 并连
接 Laboratory，但 golden SQL 直接使用 Laboratory.Date。这些例子说明，很多错误并不是 SQL 语法能力不足，而是自然语言意图、数据库结构和 benchmark 标注之间存在
欠定性。

因此，本文不应把 Text-to-SQL 简单视为“拟合唯一正确 SQL”的问题，而应将其理解为一个带有语义欠定性和评测口径约束的系统工程问题。执行准确率仍然是必要指标，但它
只能反映模型输出与 benchmark golden SQL 的一致程度，不能完全刻画模型是否理解了数据库、是否生成了业务上合理的查询，以及系统是否具备稳定的 schema grounding、
value grounding 和错误恢复能力。Pontis 的实验价值也正在于此：它不仅追求更高的 BIRD dev 准确率，还通过图谱化工作空间、工具探索、护栏机制和 token 成本统计，分析 Text-to-SQL 系统在真实数据库理解中的能力边界。


———

因此，当前 Text-to-SQL 任务的主要瓶颈未必只在方法本身，也在评测基准的建模方式上。现有评测通常采用两类指标：一类是 SQL 字符串或结构比对，另一类是执行结果比
对。前者容易将语义等价但写法不同的 SQL 判为错误，因此已经逐渐被主流 benchmark 弃用；后者通过比较 predicted SQL 与 golden SQL 的执行结果，在一定程度上缓解了
表面语法差异带来的假阴性。

但是，执行结果比对仍然没有解决前文提到的两个根本问题。首先，同一业务目标可能存在多种 SQL 实现路径，只要输出列、排序、聚合粒度或格式与 golden SQL 不一致，就
可能被判错，即使该 SQL 在真实业务场景中是合理甚至更有用的。其次，自然语言 query 本身可能是欠定的，单句问题不足以唯一决定表选择、字段来源、过滤值和聚合口
径。此时执行结果比对衡量的仍然是“是否复现了标注者选择的那一种解释”，而不一定是“是否正确理解并满足了用户真实意图”。

———

现有评测基准通常默认 Text-to-SQL 是一次性输入输出任务：用户给出一个自然语言 query，系统立即生成 SQL，并用 golden SQL 或其执行结果进行判定。但在真实数据分析
场景中，用户问题经常存在歧义或信息缺失。为了得到可靠答案，系统往往需要像 agent 一样进行追问、澄清和验证，例如确认时间范围、指标口径、输出粒度、实体定义或过
滤条件。这意味着 Text-to-SQL 不应始终被建模为单轮映射问题，而应在复杂场景下被视为一个长序列的交互式任务。

在这种设定下，用户输入不再只是一个简短 query，而是逐步澄清后形成的明确业务目标。相应地，评测标准也不应只停留在与某条 golden SQL 的字面或执行结果一致性上，
而应转向判断模型生成的 SQL 是否满足该业务目标。换言之，理想评测对象应从“SQL 是否匹配标注答案”扩展为“系统是否通过必要交互正确理解任务，并生成能够支撑业务目
标的查询”。这也更符合真实 Text-to-SQL 系统的使用方式：用户关心的不是 SQL 是否复现 benchmark 中某个唯一标注，而是查询结果是否可靠地回答了自己的业务问题。

在text2sql这个任务上,学术界和工业界在某种程度上已经产生了一定的脱节, 由于目前text2sql评测方法的不完善性,许多方法更多关注的是类似于bird数据集语法风格对齐这种对于真实业务目标毫无帮助的学术刷榜问题, 

例如bird数据集是这样划分的,他总共拥有90多个数据库,它将其中70多个数据库和其相关query划分为训练集,11个数据库划分为dev集,但问题在于,我们能从train集里面学习到什么?每个数据库都是一个独立的业务场景,我们为什么要从其他的数据库里面学习,将这种跨数据库的经验应用到当前的业务数据库上?

这本质还是我们前面说的bird数据集的语法风格问题,它对于达成用户的业务目的毫无帮助,目前的主流agent已经在单纯的sql语法生成达到了非常高的准确度,在语法方面几乎不可能出错,

因此,目前的text2sql的唯一问题其实就变成了,如何让模型更好地在和用户的交互中理解数据库结构最终达成业务目标,而仅凭bird数据集目前的评测方案以及数据集规模很难达到这点
