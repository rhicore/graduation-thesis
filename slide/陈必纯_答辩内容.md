## 第 2 页：问题背景

### 研究背景

Text-to-SQL 的核心难点，已经不只是 SQL 语法生成，而是：

- 如何理解真实数据库的结构和语义
- 如何完成稳定的 schema linking 和 value grounding
- 如何在多轮工具调用中验证假设
- 如何把数据库理解结果沉淀为可复用资产

### 研究问题

本文关注的问题是：

> 能否在原始数据与大模型之间构建一个可查询、可更新、可复用的语义层，使数据智能体先理解数据库，再生成和验证 SQL。

---

## 第 3 页：本文方法概述

### Pontis 的整体运行 pipeline

Pontis 的核心不是单次 prompt 生成 SQL，而是维护一个位于原始数据和查询智能体之间的**语义层**。

### 第一阶段：离线维护语义层

语义层由两类机制共同维护：

#### 1. 自动化脚本

负责稳定、可批量计算的信息：

- 数据库、表、列、外键等源事实
- 列统计、样例值、值分布
- 值重叠关系、AI 摘要、向量索引

#### 2. AI agent explorer

负责更难规则化的数据库理解：

- 行粒度提示
- 字段落点提示
- 局部消歧
- 数据库级 README

### 第二阶段：在线回答用户 query

当用户问题到来后，查询智能体不是直接面对原始数据库全量 schema，而是：

1. 先访问语义层
2. 再结合数据库执行验证
3. 最后完成数据分析任务并输出 SQL

## 第 4 页：实验设置与主结果

### 数据集与协议

- 数据集：BIRD dev
- 规模：11 个数据库，1534 道题
- 设置：零样本评测，不使用 BIRD train 中的 gold SQL、答案或 few-shot 示例

### 主实验结果

- **Pontis：1003 / 1534，65.38%**
- DeepEye-SQL：1001 / 1534，65.25%
- Alpha-SQL：972 / 1534，63.36%
- Bash Agent：921 / 1534，60.04%
- CHESS：900 / 1534，58.67%

### 结果解读

Pontis 在统一评测条件下：

- 相比 Bash Agent 提升明显
- 与 DeepEye-SQL 基本持平
- 说明基于语义层的工具智能体，可以达到专门 Text-to-SQL pipeline baseline 的水平

### 本文还重点分析了 token 消耗效率

除准确率外，论文还专门比较了：

- `cached input tokens`
- `uncached input tokens`
- `output tokens`
- `LLM rounds`

并进一步进行了**成本转换计算**。

### 成本转换方法

本文将三类 token 按归一化权重折算：

- cached input : uncached input : output
- `0.02 : 1 : 2`

据此计算每种方法的价格折算成本，并进一步考虑预处理摊销。

### 这一页真正要强调的结论

Pontis 的特点不是单纯 token 最少，而是：

- 稳定上下文很多，因此 cached input 很高
- 运行时新增的 uncached input 相对可控
- 在考虑缓存和预处理摊销后，总体折算成本低于 DeepEye-SQL、Alpha-SQL 和 CHESS

因此，本文实验结论不只是“谁分更高”，还包括：

> Pontis 通过把数据库理解前移到语义层维护阶段，实现了准确率、上下文复用和成本结构之间的平衡。

## 第 5 页：论文的核心结论

如果只看前六章，结论可以概括为两点：

1. Pontis 作为基于语义层的数据智能体系统，能够达到与专门 Text-to-SQL 基线相当的执行准确率。
2. Pontis 的价值不仅在于提高分数，更在于提供可审计错误归因、可缓存上下文和可增量维护的数据库语义工作空间。

但本文真正想强调的重点，其实在**第七章**：

> 当前 Text-to-SQL benchmark 的执行准确率，并不总是等价于“系统是否真正理解了用户问题”。

---

## 第 6 页：第七章的核心问题

### 为什么第七章重要

随着模型和工具能力增强，很多错误已经不再是简单的：

- SQL 不会写
- 表列找不到
- 语法生成失败

而更多变成了：

- 输出结果形态与 gold SQL 不一致
- query 本身语义欠定
- benchmark 只接受某一种标注口径

### 第七章的核心观点

当前静态 Text-to-SQL 基准主要衡量的是：

> **系统与数据集标注答案的一致性**

而不一定完全等价于：

> **系统是否满足了真实用户的业务分析目标**

---

## 第 7 页：第七章结论一：SQL 多重可达性

### 含义

同一个业务目标，往往存在多种业务上合理的 SQL 写法。

即使这些 SQL 的业务解释都说得通，只要：

- 输出列不同
- 排序不同
- 格式不同
- 数值类型不同
- 聚合粒度不同

都可能在 benchmark 中被判错。

### 这说明什么

执行准确率虽然比字符串匹配更合理，但它对**结果形态**仍然非常敏感。

因此，一部分“错误”本质上是：

> 模型没有复现 gold SQL 的输出契约，而不一定是没有理解业务目标。

---

## 第 8 页：第七章案例一

### 例子：`card_games / q390`

问题要求：找出 ID 1 到 20 的卡牌颜色和格式。

### Bash Agent 的做法

返回：

- `id`
- `colors`
- `format`

### gold SQL 的做法

只返回：

- `colors`
- `format`

### 这个例子说明

从真实分析角度看，额外返回 `id` 其实是合理的，因为它能帮助区分每张卡牌。

但在 benchmark 中，只要多输出了这一列，执行结果形状就不同，因此会被判错。

### 结论

这类错误不属于“SQL 不会写”，而属于：

> **业务上合理，但没有对齐 benchmark 的最小输出契约**

---

## 第 9 页：第七章结论二：Query 语义欠定性

### 含义

有些自然语言问题，本身并没有给出足够信息来唯一确定：

- 该选哪张表
- 该选哪个字段
- 应按什么粒度聚合
- latest / current / last 应如何解释

这时系统即使生成了一条业务上合理的 SQL，也未必与标注者选择的那条路径一致。

### 这说明什么

静态 benchmark 不允许追问和澄清，于是本来应该通过交互解决的问题，被压缩成了“一次性预测错误”。

## 第 10 页：第七章案例二

### 例子：`codebase_community / q689`

问题大意：找出 **“最后编辑 ID 为 183 的帖子的人”**。

### 两种合理理解

#### 理解 A：看编辑历史表

根据自然语言里的 `last to edit`，一种直观理解是：

- 去 `postHistory` 里找最后一条编辑记录
- 或者沿着编辑历史去找最后编辑者

这更贴近：

> “最后编辑过这篇帖子的用户是谁”

#### 理解 B：看帖子主表的编辑相关字段

另一种理解是：

- 直接读取 `posts` 表里的最后编辑字段
- 或者读取 owner / last-editor 这类角色字段

这更像是在帖子主表上寻找“相关用户”。

### gold SQL 的选择

论文中分析到，这道题的 gold SQL 最终连接的是 **owner 字段**，而不是严格意义上“最后编辑记录”对应的路径。

### 这个例子说明

自然语言中的 `last to edit`，和数据库里的：

- `postHistory`
- `LastEditorUserId`
- `OwnerUserId`

并不是天然一一对应的。

如果题面和 evidence 没有进一步澄清，系统很难仅凭一句 query 唯一确定到底该走哪条路径。

因此，这类错误的根源不是纯粹的数据库探索不足，而是：

> **自然语言 query 本身存在语义欠定**

---

## 第 11 页：第七章带来的总体认识

第七章最终想说明三点：

### 1. 静态执行准确率是必要指标，但不是充分指标

它适合做统一、可复现的横向比较，但不能完全等价于真实业务正确性。

### 2. Text-to-SQL 错误来源是混合的

错误可能同时来自：

- 数据库探索不足
- 输出形态不一致
- 标注口径偏好
- query 语义欠定

### 3. 更合理的未来方向是交互式评测和 query log 评测

真实数据智能体应当能够：

- 追问用户澄清口径
- 记录历史经验
- 在同一数据库中长期复用理解结果
