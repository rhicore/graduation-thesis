\begin{abstractcn}

随着企业和科研场景中的数据项目不断扩展，
关系数据库、表格文件、JSON 文档和说明文档常常以松散目录的形式共同存在。
大语言模型虽然具备较强的自然语言理解和代码生成能力，
但在直接面对真实数据项目时仍存在上下文容量有限、数据库模式关系隐式、领域术语歧义以及分析经验难以复用等问题。
针对这些问题，本文设计并实现了 Pontis：一种面向项目数据的图谱化智能体工作空间。
Pontis 不直接训练或微调专用的 Text-to-SQL 模型，而是在原始数据与大语言模型智能体之间引入可查询、可更新、可复用的知识图谱中间层，
使智能体能够先理解数据项目，再执行自然语言查询和数据分析任务。

本文首先提出了以知识图谱为核心的项目工作空间架构，
将文件、数据库、表、列、外键、知识经验等语义单元统一建模为图实体及关系。
系统通过模块化脚本在预处理和预查询阶段为图节点
生成列统计、值分布、列重叠、外键验证、语义摘要和向量索引等元数据和派生知识，
并通过统一的图接口向上层工具和智能体提供图查询能力。
其次，本文设计了面向数据分析的工具调用智能体，
支持为其提供了十余种图谱及数据探索工具，
并引入轮次限制、探索检查、SQL 实体检查、桥接表检查和 SQL 消歧检查等护栏机制，
以降低幻觉、错误连接和过度探索风险。

% ---------- 英文摘要 ----------
\begin{abstracten}

As data projects in enterprise and research scenarios continue to grow, relational databases, tabular files, JSON documents, and project documentation often coexist in loosely organized directories. Large language models have shown strong capabilities in natural language understanding and code generation, but they still struggle with real data projects because of limited context windows, implicit schema relationships, ambiguous domain terminology, and the lack of reusable analysis experience. To address these problems, this thesis designs and implements Pontis, a graph-native agent workspace for project data. Instead of training a specialized Text-to-SQL model, Pontis introduces a queryable, updatable, and reusable knowledge graph layer between raw data and LLM agents, enabling agents to understand a data project before answering natural language queries.

This thesis first presents a workspace architecture centered on Neo4j and Cypher. Files, databases, tables, columns, foreign keys, statistical summaries, semantic descriptions, disambiguation hints, and cross-project experience are modeled as graph entities and relationships. Source modules publish source-derived facts such as file-system and SQLite schema information before graph queries. Extractor modules generate derived knowledge including column statistics, value distributions, column overlaps, foreign-key validation, semantic summaries, and vector indexes. All upper-level tools and agents access the graph through a unified \texttt{Workspace.cypher} interface. The thesis then presents a tool-calling data agent equipped with tools such as \texttt{find}, \texttt{meta}, \texttt{query}, \texttt{grep}, \texttt{read}, and \texttt{cypher}. Guardrails including round limitation, exploration checking, SQL entity checking, bridge-table checking, and SQL disambiguation checking are introduced to reduce hallucination, incorrect joins, and unnecessary exploration.

Preliminary experiments on the BIRD Text-to-SQL development set show that Pontis achieves 64.45\% execution accuracy on 1,533 currently logged questions. Error analysis indicates that the major failure sources are BIRD-specific SQL style preferences, local schema misunderstanding, value grounding errors, and join-path mistakes. The planned full evaluation compares Pontis with Bash Agent, CHESS, Alpha-SQL, and DeepEye-SQL under a shared model configuration, private data copies, and a unified timing protocol. The comparison focuses on execution accuracy, preprocessing cost, online latency, LLM calls, input tokens, tool calls, and SQL executions. The value of Pontis is not to replace specialized SQL models, but to provide a transparent, cacheable, and continuously evolving data-understanding workspace for general-purpose data agents.

\end{abstracten}

\section{绪论}

\subsection{研究背景与意义}

数据分析任务正在从单一数据库查询转向面向整个数据项目的综合理解。一个真实数据项目中通常同时包含关系数据库、CSV 或 TSV 表格、JSON 配置、Markdown 说明、实验日志、历史查询以及人工整理的业务规则。传统数据库系统强调数据的存储、索引和查询效率，但并不负责把这些分散的项目知识组织成适合自然语言智能体使用的上下文。Text-to-SQL 技术试图将自然语言问题转换为 SQL 查询，从而降低普通用户访问结构化数据的门槛；但当任务从教学型数据库走向真实项目目录时，系统面临的核心问题已经不只是“如何写出一条 SQL”，而是“如何让智能体先理解一个数据项目”。

近年来，大语言模型在 Text-to-SQL 任务中取得了显著进展。Spider\upcite{yu2018spider} 推动了跨数据库泛化研究，BIRD\upcite{li2023bird} 进一步引入大规模真实数据库、数据库值、外部 evidence 和 SQL 执行效率要求，使任务更接近真实分析场景。随后出现的 Spider 2.0\upcite{lei2025spider2}、BEAVER\upcite{chen2025beaver}、Falcon\upcite{luo2025falcon} 和 SQaLe\upcite{wolff2025sqale} 等基准表明，真实数据环境往往具有更庞大的 schema、更复杂的业务语义、更长的上下文和更不稳定的数据文档。InsightBench\upcite{sahu2025insightbench} 进一步把目标从单条 SQL 扩展到多步业务洞察，说明自然语言数据分析已经开始超出传统 Text-to-SQL 的边界。企业内部系统的实践研究也显示，有效的 Text-to-SQL 系统需要索引数据库元数据、历史查询、文档和代码等多源知识，而不能只依赖表名和列名\upcite{chen2025enterpriseTextToSQL,johnson2024querygpt}。

因此，直接将大语言模型用于真实数据项目仍面临以下困难。

第一，数据库模式和项目上下文过大。真实数据库可能包含大量表、列、视图和项目文档。若将完整 schema、样例值和说明文档直接塞入提示词，会造成上下文冗长、成本上升和注意力分散；若预先过滤 schema，又可能错误丢弃关键表列。尽管强模型在小规模 schema 中可以输入更多上下文，但大规模企业 schema 仍然需要可靠的上下文控制机制，来控制模型的认知复杂度\upcite{maamari2024deathSchemaLinking,nahid2025schemaLinking,eben2025rasl}。

第二，关系、语义和值口径常常是隐式的。许多数据库没有完整外键约束，表间连接依赖命名约定、值域重叠、历史查询或项目说明。列名相同并不意味着语义相同，列名不同也可能表达相同业务概念；即使表列选择正确，过滤值、枚举别名、日期边界和聚合口径也仍可能出错。自动元数据抽取、schema graph 和 schema refinement 相关研究表明，列统计、值分布、查询日志、业务规则和 LLM 生成的字段描述可以显著补足人工文档缺失的问题\upcite{shkapenyuk2025metadata,wang2025schemaGraphQA,rissaki2024schemaRefinement}。

第三，模型必须在真实数据上验证假设。自然语言问题中的过滤值、枚举项、日期边界、比例分母和聚合粒度往往无法仅靠 schema 推断。SQL-PaLM、CHESS、ReFoRCE、APEX-SQL 和 DeepEye-SQL 等系统分别通过执行反馈、候选选择、多智能体协作、列探索和软件工程式编排来提升可靠性\upcite{sun2024sqlpalm,talaei2024chess,deng2025reforce,cao2026apexsql,li2025deepeyesql}。这些工作说明，Text-to-SQL 正在从一次性生成转向带工具调用、检索、验证和修正的 test-time framework。

第四，评测结果本身需要谨慎解释。BIRD 等基准中存在自然语言歧义、gold SQL 口径差异和标注错误；已有研究发现，噪声和标注问题会影响模型排名和方法判断\upcite{wretblad2024birdNoise,jin2026annotationErrors}。这意味着系统评估不能只报告最终执行准确率，还应分析错误来源、工具调用成本、预处理成本和在线查询延迟。

本文认为，真实数据项目中的自然语言分析不应只被视为单次 SQL 生成问题，而应被视为“智能体如何持续理解并维护一个数据项目”的问题。Pontis 因此在原始数据和 LLM 智能体之间引入图谱化工作空间，将 schema、统计、关系、摘要、消歧和经验统一表示为可查询知识，使智能体能够通过工具逐步获取精确上下文，而不是每道题从零扫描整个项目。

\subsection{主要贡献}

围绕上述问题，本文设计并实现了 Pontis 图谱化数据工作空间。本文的主要贡献如下：

（1）提出一种图谱化数据工作空间架构。该架构以 Neo4j 为持久化图存储，以 Cypher 为统一查询接口，以自动化脚本和智能体探索共同维护项目知识，避免将全部数据粗暴拼接到 LLM 上下文中。

（2）实现一套面向数据库项目的知识提取与组织机制。系统能够自动暴露文件系统、SQLite 模式、表列结构和外键关系，并进一步生成列级统计、值样例、列重叠、AI 摘要和语义向量等派生知识。

（3）设计了结合工具调用和护栏的 LLM 数据智能体。系统将图谱检索、元数据查看、SQL 执行、文件搜索、JSON 查看和图写入操作封装为工具，并通过 SQL 实体检查、桥接表检查和消歧检查减少常见数据库问答错误。

（4）给出面向 test-time framework 的实验设计。本文将 Pontis 与 Bash Agent、CHESS、Alpha-SQL、DeepEye-SQL 等方法放在统一评测协议下比较，强调准确率之外的系统性能指标和错误归因，使系统型贡献能够被量化分析。

\subsection{论文结构}

本文结构如下：第二章介绍相关工作；第三章给出 Pontis 的系统架构；第四章阐述自动化图谱构建与智能体探索式知识沉淀机制；第五章介绍智能体工具系统与护栏机制；第六章给出实验设计、阶段性结果和性能分析；第七章总结全文并展望未来工作。

\section{相关工作}

\subsection{Text-to-SQL 基准与真实场景演进}

Text-to-SQL 是自然语言处理与数据库系统交叉领域的重要任务，其目标是将自然语言问题转换为可执行 SQL。早期方法多依赖规则和语义解析，随后 Seq2SQL\upcite{zhong2017seq2sql} 等方法将神经网络和强化学习引入 SQL 生成。Spider\upcite{yu2018spider} 的提出推动了跨数据库泛化研究，RAT-SQL\upcite{wang2020rat} 通过关系感知编码器建模问题与数据库 schema 之间的关系，代表了在深度模型时代显式建模 schema linking 的典型路线。

大语言模型出现后，Text-to-SQL 的研究重心从模型结构设计逐渐转向提示工程、检索增强、任务分解和执行反馈。C3\upcite{gao2024c3} 和 DAIL-SQL\upcite{gao2023dailsql} 系统比较了提示表示、示例选择和自修正等因素，DIN-SQL\upcite{pourreza2024dinsql} 则将复杂 SQL 生成拆分为 schema linking、查询分类和 SQL 生成等子任务。这些工作表明，强 LLM 不一定需要任务级微调才能完成复杂 SQL，但其效果高度依赖上下文组织和推理流程设计。BIRD\upcite{li2023bird} 进一步引入更大的数据库、外部 evidence、真实值理解和 SQL 效率要求，使 Text-to-SQL 从语法映射转向数据库内容理解。

近年的基准进一步强调企业级复杂性和端到端数据分析能力。Spider 2.0\upcite{lei2025spider2} 将任务扩展到真实数据应用和 SQL workflow，BEAVER\upcite{chen2025beaver} 从企业数据仓库查询日志中构造评测集，Falcon\upcite{luo2025falcon} 关注中文企业语义和工业方言，SQaLe\upcite{wolff2025sqale} 则在真实 schema 上构造更大规模语料。InsightBench\upcite{sahu2025insightbench} 将评测对象从单条 SQL 扩展到多步业务洞察生成，ExeSQL\upcite{zhang2025exesql} 关注执行驱动的 SQL 方言适配。这些工作共同说明，真实 Text-to-SQL 系统需要处理远大于教学数据集的 schema 空间、业务术语、私有规则、SQL 方言和多轮分析流程。本文关注的 Pontis 正是沿着这一趋势，把 Text-to-SQL 问题放到“项目级数据理解”的系统环境中讨论。

\subsection{Schema Linking 与元数据增强}

Schema linking 负责将自然语言问题中的实体、属性和约束映射到数据库表列，是 Text-to-SQL 系统的核心环节。传统方法通常将 schema linking 作为 SQL 生成前的过滤步骤，但近期研究对这一环节提出了更细的分析。Maamari 等人的研究指出，当完整 schema 能放入上下文且模型推理能力足够强时，过早过滤 schema 可能因为召回错误而伤害生成结果\upcite{maamari2024deathSchemaLinking}；Nahid 等人和 RASL 则从相反方向说明，面对大规模企业 schema 时，全量上下文会带来无关信息、token 成本和幻觉风险，因此仍然需要上下文感知和检索增强的 schema linking\upcite{nahid2025schemaLinking,eben2025rasl}。

单纯依赖表名和列名也不足以解决真实场景中的语义问题。自动元数据抽取研究从数据库 profiling、列级统计、值样例、历史查询日志和 LLM 生成字段摘要中提取补充信息\upcite{shkapenyuk2025metadata}；schema refinement 工作则把原始数据库整理为更易解释的语义层，降低用户问题与底层 schema 之间的语义距离\upcite{rissaki2024schemaRefinement}。面向多表问答的 schema graph 方法进一步说明，表列之间的连接关系、候选路径和人工确认信息适合用图结构表达，而不应完全依赖模型在生成时临场推断\upcite{wang2025schemaGraphQA}。

更进一步，schema linking 只解决“去哪些表列查”的问题，并不能自动保证过滤值、枚举映射、日期边界和聚合口径正确。BIRD 的设计已经强调数据库值和外部 evidence 的作用\upcite{li2023bird}，而企业系统中常见的业务简称、历史口径和隐式指标定义则使 value grounding 更加困难。RubikSQL 将 NL2SQL 视为长期学习任务，持续维护业务术语、规则和失败经验\upcite{chen2025rubiksql}；LinkedIn 的企业 Text-to-SQL 系统也把元数据、查询日志、文档和代码共同纳入知识图谱\upcite{chen2025enterpriseTextToSQL}。这些工作表明，真实数据项目中的上下文不只是 schema 文本，而是 schema、值、关系和经验的组合。

Pontis 与这些工作的共同点在于重视元数据、值分布和关系知识；不同点在于，Pontis 将这些信息统一落到项目级图谱工作空间中，并通过工具接口让智能体按需读取，而不是一次性把全部增强 schema 拼入 prompt。

\subsection{智能体式 Text-to-SQL 与执行验证}

工具调用是增强 LLM 能力的重要途径。Toolformer\upcite{schick2023toolformer} 探索了语言模型如何学会使用外部 API，ReAct\upcite{yao2023react} 将推理和行动交替组织，使模型能够在环境反馈中逐步完成任务。在 Text-to-SQL 中，工具调用可以弥补模型静态知识不足的问题：模型可以执行 SQL 验证假设，通过搜索定位字段，通过样例值确认枚举，通过错误反馈修正语法和逻辑。

围绕 BIRD 和 Spider 2.0，已有方法逐渐形成多阶段、可验证的 test-time framework。SQL-PaLM\upcite{sun2024sqlpalm} 研究了 few-shot prompting、执行过滤、consistency decoding、指令微调和数据库内容使用对性能的影响；CHESS\upcite{talaei2024chess} 使用信息检索、schema 选择、候选生成和单元测试等多个 agent 分工协作；ReFoRCE\upcite{deng2025reforce} 引入表压缩、格式约束、自修正、共识选择和列探索；APEX-SQL\upcite{cao2026apexsql} 将生成过程组织为假设验证循环，通过真实数据探索落地语义；DeepEye-SQL\upcite{li2025deepeyesql} 则将 SQL 生成类比为软件开发过程，通过需求分析、候选实现、测试和发布选择提升可靠性。上述工作说明，复杂 Text-to-SQL 的关键并不只是更强的单次生成，而是如何组织检索、探索、验证和选择。

除显式工具调用外，候选集选择和 test-time scaling 也是提升稳定性的常见路径。MCS-SQL\upcite{lee2024mcssql} 通过多提示生成候选并进行选择，self-consistency\upcite{wang2023selfConsistency} 说明多条推理路径的一致性可以提高复杂推理可靠性，Agentar-Scale-SQL\upcite{wang2025agentarScaleSQL} 进一步讨论了编排式 test-time scaling 在 Text-to-SQL 中的作用。RoboPhD\upcite{borthwick2026robophd} 从自演化 agent 角度探索系统如何根据历史失败持续改进。这类方法提升了复杂任务上的准确率，但通常也带来更多 LLM 调用、SQL 执行和工具轮次，因此需要和系统性能指标一起分析。

Pontis 也采用工具智能体范式，但其重点不是构造更多候选 SQL，而是提供一个可持久化的数据理解层。智能体的每次检索、元数据查看和 SQL 验证都围绕图实体展开，后续问题可以复用已有的统计、摘要、消歧和经验知识。

\subsection{知识图谱与数据项目理解}

知识图谱将实体及其关系表示为图结构，适合表达数据库 schema、文件组织、表列归属、外键关系和业务概念之间的联系。传统知识图谱构建强调从文本中抽取实体和关系，而数据项目场景中的图谱构建还需要处理结构化源事实和统计派生知识。企业 Text-to-SQL 实践已经显示，知识图谱可以把数据库元数据、历史查询、文档和代码等多源信息组织为可检索上下文，为 SQL 生成和错误修复提供依据\upcite{chen2025enterpriseTextToSQL}。

在企业数据分析场景中，知识图谱的价值不仅在于存储概念关系，还在于为智能体提供可操作的项目地图。Schema graph 相关工作使用图结构减少模型对隐式连接关系的猜测\upcite{wang2025schemaGraphQA}；RubikSQL 将长期运行中沉淀的业务知识、错误修正和查询经验组织为可复用知识库\upcite{chen2025rubiksql}；LinkedIn 的企业系统使用知识图谱统一数据资产、元数据和查询经验\upcite{chen2025enterpriseTextToSQL}；Uber QueryGPT 则结合向量数据库和相似查询检索，将自然语言问题和内部数据模型知识连接起来\upcite{johnson2024querygpt}。这些实践说明，Text-to-SQL 系统需要一种位于数据库和模型之间的中间表示，用来承接不断变化的项目知识。

Pontis 的图谱并不是静态业务知识库，而是围绕项目数据持续生成和更新的工作空间。文件系统和数据库 schema 可以作为源事实按需刷新，AI 摘要、列统计、列重叠和人工经验可以作为派生事实持久化。通过这种方式，Pontis 将“当前项目是什么”和“过去分析学到了什么”统一到一个可查询图中。与只服务于单次 SQL 生成的增强 prompt 相比，图谱工作空间更适合跨问题复用、错误归因和后续知识更新。

\subsection{评测噪声与系统指标}

Text-to-SQL 评测通常依赖执行准确率，即比较预测 SQL 与参考 SQL 的执行结果是否一致。然而，执行结果并不总能完整反映语义正确性：不同 SQL 可能因输出列、排序、聚合口径或边界条件差异而产生不同结果；参考 SQL 本身也可能存在标注错误或问题歧义。针对 BIRD 的噪声分析指出，问题和 gold SQL 中的错误会影响模型表现和结论可靠性\upcite{wretblad2024birdNoise}；进一步的 benchmark 审计研究也表明，标注错误可能显著改变模型排名\upcite{jin2026annotationErrors}。

随着基准复杂度提高，评测对象也从“单条 SQL 是否匹配”扩展为“系统是否能稳定完成分析任务”。针对 Text-to-SQL benchmark 的错误分析指出，数据集中的自然语言歧义、参考 SQL 口径和结果比较方式都会影响 leaderboard 结论\upcite{jin2026benchmarkErrors}。因此，系统比较需要同时报告错误类型和资源消耗，尤其要区分 schema linking 失败、value grounding 失败、SQL 语义错误、输出口径差异和评测噪声。

因此，本文在评估 Pontis 时不仅关注执行准确率，还关注预处理成本、在线查询耗时、LLM 调用次数、工具调用次数、SQL 执行次数和错误归因。这样的系统指标更适合衡量图谱化工作空间的价值：即它是否能减少重复探索、压缩上下文、提高 schema/value grounding 的可靠性，并为后续查询积累可复用知识。

\subsection{总体设计}

Pontis 的核心思想是在原始数据源和智能体之间建立图谱化中间层。系统整体由四个部分组成：原始数据源、图谱工作空间、知识提取器和工具智能体。其数据流如图\ref{fig:architecture_text} 所示。

  -> LLM Agent

\end{verbatim}

原始数据源包括本地文件、SQLite 数据库、CSV/TSV 表格、JSON 文档和文本说明。Pontis 首先通过确定性的自动化脚本将这些数据源中的基础事实发布为图节点和边，例如目录、文件、数据库、表、列和外键；随后继续在图谱之上生成统计、摘要、关系和向量等增强信息。Neo4j 图谱工作空间负责持久化这些事实和派生知识，并提供 Cypher 查询接口。工具智能体通过工具访问图谱和原始数据，逐步完成自然语言分析任务。

系统中最重要的边界是 \texttt{Workspace.cypher(...)}。所有上层工具和自动化提取脚本都通过该接口访问图谱，而不是直接耦合到具体存储实现。这样可以将数据源刷新、图查询、结果指针解析和模块触发逻辑集中在工作空间层，降低工具层复杂度。

\subsection{图谱数据模型}

Pontis 将项目数据表示为属性图。主要实体类型包括：

（1）文件实体。表示项目目录中的文件，包含路径、文件类型和源访问指针等属性。

（2）数据库实体。表示 SQLite 等数据库文件，作为表、视图和列的上级节点。

（3）表与视图实体。表示数据库表、视图或表格文件中的逻辑表，记录名称、来源和结构信息。

（4）列实体。表示表中的字段，记录列名、类型、样例值、统计信息和语义描述。

（5）外键和关系实体。表示显式外键、推断关系、列重叠和消歧知识。

（6）知识实体。表示人工或自动沉淀的规则、经验、错误教训和 benchmark 案例。

Pontis 的建模原则是：能够用边表达的归属和来源关系，不重复写入节点属性。例如，表属于数据库、列属于表、chunk 来源于文件，都优先用图边表示。节点属性只保存实体自身的局部事实，如列类型、摘要文本、统计量和向量索引状态。这一原则可以减少事实重复，避免移动或更新数据源时产生不一致。

\subsection{Neo4j 与 Cypher 接口}

Pontis 使用 Neo4j 作为当前主图存储。工作空间层只暴露 Cypher 接口，基础事实发布脚本和工具通过生成 Cypher 语句完成图实体写入、匹配和查询。使用 Neo4j 的好处在于：

（1）Cypher 能直接表达图模式匹配、邻居遍历和多跳关系查询，适合表列关系、外键关系和知识实体检索。

（2）图存储与工具层解耦。工具只需要关心实体 ref、标签和查询结果，不需要自己维护图索引。

（3）跨项目路由更清晰。Pontis 可以为不同项目配置不同 Neo4j 端口或数据库，使同一套工具能够查询多个项目图。

（4）后续可利用 Neo4j 的索引、向量检索和查询优化能力，支撑更大规模数据项目。

\section{图谱构建与知识提取机制}

第四章介绍 Pontis 如何把原始项目数据转化为可查询、可复用的图谱知识。本章不将 store module、提取器和 Explorer 划分为业务层面的三套系统；它们只是代码实现上的分工。业务上，Pontis 要完成的是同一件事：持续构建数据项目知识图谱。实现上，一部分知识由确定性的自动化脚本生成，例如文件、表、列、外键、统计、JSON 模式、列重叠和向量索引；另一部分知识由智能体在工具环境中探索、打标和总结，例如连接经验、字段消歧、表职责和局部约定。两类机制共享同一个图谱工作空间，使基础结构、派生特征和经验知识能够被工具智能体统一读取。

Pontis 的图谱构建并不是一次性离线导入，而是“按需发布 + 批量提取 + 智能体探索”的组合过程。按需发布保证查询前能够看到文件、表、列和外键等基础实体；批量提取负责补充统计、值分布、JSON 模式、列重叠、摘要和向量；智能体探索则处理更依赖语义判断的连接关系和消歧知识。这样的设计使系统可以把确定性事实、可计算派生知识和 LLM 判断结果分开管理，降低错误知识污染图谱的风险。

\subsection{基础事实的按需发布}

基础事实发布机制负责把原始数据源暴露为可查询图。它不直接回答用户问题，也不执行复杂分析，而是把文件系统和数据库 schema 这样的源事实转化为图节点和边。当前文件系统项目的典型发布单元包括：

  \centering
  \topcaption{Pontis 基础事实发布单元}
  \label{tab:source_modules}
  \begin{tabularx}{\textwidth}{lX}
    \toprule[1.5pt]
    \textbf{模块} & \textbf{职责} \\
    \midrule[1pt]
    文件系统模块 & 发布目录与文件实体 \\
    文本文件模块 & 发布文本文件元信息和读取端口 \\
    表格结构模块 & 发布 CSV/TSV 表结构 \\

工作空间在执行 Cypher 前会解析查询涉及的标签和属性，并根据触发规则选择需要刷新的发布单元。例如，当查询涉及 \texttt{table}、\texttt{col} 或 \texttt{fk} 标签时，数据库结构单元会先发布数据库结构；当查询只涉及文本文件时，则不必刷新数据库单元。对于读查询，发布结果会被缓存；对于写查询，系统会在执行后使相关缓存失效，以保证后续读取的一致性。该机制避免了每次启动都扫描全部数据源，也使工具可以用同一套 Cypher 接口查询不同粒度的项目实体。

按需发布的执行路径如下：

\begin{equation}
Q \rightarrow I(Q) \rightarrow M_Q \rightarrow G_s \rightarrow G_q ,
\end{equation}

其中 $Q$ 表示原始 Cypher 查询，$I(Q)$ 表示对查询中标签、属性和读写类型的轻量解析，$M_Q$ 表示被触发的基础事实发布单元集合，$G_s$ 表示发布到 Neo4j 的源事实子图，$G_q$ 表示最终被原始查询读取的图谱视图。发布单元只提交由自己生成的 Cypher 写入语句，具体的查询执行和结果返回仍由 Neo4j 完成。这样做的好处是，存储层不需要实现第二套图查询引擎，各发布单元也不需要了解彼此的内部状态。

\subsection{节点身份、边与访问指针}

Pontis 在图谱中同时维护“实体身份”和“原始数据访问能力”。实体身份由路径、引用和内部 id 共同保证：文件节点通常以路径合并，表和列节点使用数据库路径、表名和列名组成稳定引用，外键和关系实体则通过两端表列确定。所有节点都会获得存储层生成的内部 id，用于稳定边连接和工具显示。

图谱建模遵循一个原则：能够用边表达的归属和来源关系，不重复写入节点属性。例如，表属于数据库、列属于表、JSON 模式来源于 JSON 文件、文本 chunk 来源于文本文件，都应通过图边表达；节点属性只保存实体自身的局部事实，如列类型、样例值、统计量、摘要文本和向量状态。这个原则减少了重复事实，也避免了移动文件或刷新 schema 时出现“边已更新但属性仍指向旧来源”的不一致。

对于原始数据访问，Pontis 使用 resolver pointer 机制。Neo4j 中保存的只是普通字符串形式的访问指针，例如文件打开端口或数据库连接端口；当 \texttt{Workspace.cypher(...)} 返回结果后，工作空间再根据指针中的 project、module、kind 和 payload 调用对应模块的解析函数，把字符串替换为运行时 Python 对象。这样既保留了 Neo4j 作为唯一图存储和 Cypher 执行面，又让上层工具能够通过节点访问原始文件和 SQLite 连接。

\subsection{自动化脚本式知识提取}

自动化脚本式知识提取负责生成可以稳定计算或批量总结的图谱知识。实现上，每个提取单元接收一个 \texttt{Workspace} 对象，通过 Cypher 和工具函数读取源事实，并将派生结果写回图谱。它覆盖的任务既包括对原始数据的结构化扫描，也包括在基础事实之上生成统计、摘要、关系候选和向量索引。因此，这类脚本可以离线批量运行，也可以按实验需要只运行其中一部分。

自动化提取框架具有三个特点。第一，模块粒度较细，可以根据实验需要选择只运行统计、摘要或语义向量模块。第二，模块执行顺序可配置，既可以先运行轻量统计和关系发现，再运行 LLM 摘要，也可以在 benchmark 中跳过高成本模块。第三，提取结果回写到同一个图谱工作空间中，后续工具和智能体无需理解模块内部实现，只需查询图实体。

在典型数据库项目中，自动化流水线可以分为四个阶段。第一阶段读取表、列和外键等基础节点，建立基础图；第二阶段扫描真实数据，生成列统计、样例值、高频值和 JSON 模式；第三阶段根据外键、值域重叠和统计特征生成关系候选；第四阶段使用 LLM 或 embedding 服务生成语义摘要和向量索引。各阶段的输出都以节点属性或派生实体的形式写回图谱，因此后续阶段可以复用前序阶段的结果。

\subsection{列统计与值分布}

列统计是 Text-to-SQL 中的重要上下文。仅靠列名往往无法判断一个字段的真实含义，例如 \texttt{type}、\texttt{id}、\texttt{status} 等列名在不同表中可能含义不同。Pontis 对数据库和表格文件中的列提取以下信息：

（1）基本类型信息，包括 SQLite 声明类型和归一化类型。

（2）空值、不同值数量、样例值和文本长度等轻量统计。

（3）数值列的范围和分布摘要。

（4）文本列的高频值，用于帮助模型理解枚举字段和业务取值。

这些统计信息会写入列实体的元数据，并在 \texttt{meta} 工具中呈现。与直接把整列数据交给 LLM 不同，统计摘要将大量原始值压缩成稳定、低成本的上下文。对于大列，系统不要求保存完整 distinct 集合，而是使用近似基数估计和流式高频项统计来降低预处理成本；对于文本列，系统保留少量去重样例和 top-k 高频值，使智能体能够判断枚举值、编码格式和自然语言问题中的值映射。

列统计同时服务于性能和准确率。性能上，它避免智能体为了理解一个枚举列而反复执行 \texttt{SELECT DISTINCT}；准确率上，它帮助模型判断过滤条件是否需要大小写转换、是否存在缩写、是否应该使用编码列还是描述列。许多 value grounding 错误并不是模型找不到表列，而是没有看到列中的真实取值。列统计把这类证据提前压缩到图谱中，降低在线查询轮次。

\subsection{智能体探索式知识沉淀}

真实数据库中并非所有连接关系都有显式外键。Pontis 因此同时利用三类信息发现关系：

（1）数据库内声明的外键。数据库结构模块读取外键定义并将其建模为 \texttt{fk} 实体。

（2）列值重叠。自动化脚本对不同表的候选列计算值域交集和相似度，生成 \texttt{overlap} 实体，提示可能存在的 join 路径。

（3）智能体探索。智能体可以通过读取 schema、样例值和执行查询，生成更高层的 join 经验和消歧实体。

列值重叠模块采用漏斗式筛选。系统先用表名和列名 token 做表级上下文过滤，避免在完全无关的表之间枚举所有列对；再读取两列非空 distinct 值，计算交集、Jaccard 相似度和双向覆盖率；最后根据列名 token 是否共享，将候选分为强匹配和弱匹配，并为每对表只保留少量最高分候选。这样生成的 \texttt{overlap} 不是最终 join 规则，而是结构化线索，供智能体探索流程和最终问答智能体进一步验证。

智能体探索流程负责处理脚本难以完全决定的高层知识。连接发现任务会读取外键、overlap、列统计和样例值，只在满足高置信度时创建 \texttt{rel} 实体；消歧任务会扫描同名或近名列，基于样例值和表职责判断语义差异，创建 \texttt{disambig} 实体并补充相关列的 detail。这类流程的提示词强调“宁缺毋滥”：错误关系比缺失关系更危险，因为下游智能体可能把错误 \texttt{rel} 当成 join 证据。

智能体沉淀的消歧知识主要用于处理同名字段和领域术语。例如多个表中都存在 \texttt{id}、\texttt{name} 或 \texttt{type} 时，智能体在最终生成 SQL 前需要确认字段来源。Pontis 将这些消歧提示作为图实体保存，并通过 SQL 消歧护栏在高风险 SQL 输出时提醒模型读取相关信息。与一次性 prompt 不同，这些探索生成的知识会留在图中，后续问题可以直接复用。

\subsection{语义摘要与向量索引}

Pontis 中的 \texttt{brief} 和 \texttt{detail} 是面向智能体的核心文本属性。统计模块能告诉模型“这一列有什么值”，AI 摘要模块进一步解释“这一列代表什么语义”。系统会对表、列、JSON 和文本文件生成摘要，并通过语义向量模块为 detail 文本生成向量。

列摘要模块在生成列说明时会输入列名、类型、基数、空值比例、数值范围、样例值和高频值，并要求输出稳定的业务语义而不是易过时的具体统计数字。对于表摘要，系统会聚合同表列信息和外键关系，描述表的职责、粒度和常见关联路径。对于 JSON 和文本文件，系统会提取结构模式、字段分布或文本分片摘要，使非关系型文件也能进入同一个检索空间。

语义向量模块只对存在非空 \texttt{detail} 的节点生成 embedding，并记录模型名、向量维度和 detail 文本 hash。当 detail 未变化且 embedding 配置一致时，模块会跳过该节点，避免重复请求 embedding 服务。向量写回 Neo4j 后，系统按节点标签创建向量索引，使 \texttt{find(ref, query)} 不再只是名称匹配，而可以根据自然语言问题检索相关实体。例如，当问题涉及“percentage”“ratio”“foreign player”或“school SAT score”时，系统可以召回对应表列、知识规则或历史案例。这样可以减少全量浏览 schema 的成本，使智能体更快聚焦到候选实体。

\subsection{增量刷新与成本控制}

图谱构建的成本主要来自三部分：源扫描、数据统计和 LLM/embedding 调用。Pontis 对这三部分分别采用不同策略。源扫描通过数据源 fingerprint 和刷新缓存避免重复发布未变化的数据源；数据统计尽量使用单次扫描、近似基数和 top-k 结构压缩原始值；LLM 摘要和语义向量则通过缺失字段检查、文本 hash 和批量请求减少重复调用。

这种设计使 Pontis 能在预处理成本和在线查询成本之间做明确取舍。预处理阶段多花费一些时间生成统计、摘要和关系线索，可以减少智能体在线阶段的工具调用轮次、SQL 探索次数和输入 token。另一方面，并非所有项目都需要运行完整流水线：在轻量场景中可以只运行基础事实发布和列统计；在复杂 Text-to-SQL benchmark 中，则可以追加列重叠、AI 摘要、向量索引和智能体探索，以换取更强的 schema linking、value grounding 和 join-path 可靠性。

\section{工具智能体设计}

\subsection{智能体循环}

Pontis 智能体采用工具调用循环。每轮循环中，系统将用户问题、系统提示、工具定义和历史消息发送给 LLM；若 LLM 返回工具调用，Pontis 执行对应工具并把结果作为下一轮输入；若 LLM 返回最终文本，则经过护栏检查后输出结果。

该过程可以表示为：

\begin{equation}
A_t = \mathrm{LLM}(Q, H_{t-1}, T),
\end{equation}

其中 $Q$ 表示用户问题，$H_{t-1}$ 表示历史上下文和工具结果，$T$ 表示可用工具集合，$A_t$ 表示第 $t$ 轮模型动作。若 $A_t$ 是工具调用，则系统执行工具并更新 $H_t$；若 $A_t$ 是最终 SQL 或自然语言答案，则结束循环。

工具智能体的在线性能主要受三类因素影响：串行 LLM 轮次、工具执行时间和工具结果回填后的输入 token。一次 LLM 轮次可能调用多个工具，因此工具调用次数不能直接等同于 LLM 往返次数。本文在实验中分别统计 \texttt{LLM Rounds/Q}、\texttt{Tool Calls/Q} 和 \texttt{Input Tokens/Q}，以区分串行等待、探索操作量和上下文膨胀。

\subsection{提示组装}

Pontis 的提示词不是单个固定 prompt，而是由多个片段按运行模式组合而成。只读模式通常包含基础身份、工具说明、图谱本体、元数据约定、SQL 规范、护栏规则、只读约束、项目上下文和 README 摘要等片段。写入模式会额外说明如何创建知识实体和维护图关系；反思模式则强调错误归因和经验沉淀。

这种分层组装方式有两个优点。第一，系统可以针对不同任务限制工具权限，减少模型误用写入工具的风险。第二，实验可以通过开关 prompt 片段和工具集合进行消融分析，例如禁用 BIRD global 知识、禁用 query 工具或禁用消歧实体。

\subsection{护栏机制}

Pontis 的护栏层位于 LLM 输出和工具执行之间。每个护栏独立检查当前 pending tool calls 或最终文本输出，并给出 allow、warn 或 block 裁决。多个护栏的裁决按 block 高于 warn 高于 allow 的优先级聚合。

\section{实验设计与阶段性结果}

\subsection{研究问题}

本文实验围绕以下问题展开：

（1）Pontis 相比只给 shell/sqlite/file 工具的 Bash Agent，是否能通过图谱化数据工作空间提升 Text-to-SQL 准确率和稳定性？

（2）Pontis 与 CHESS、Alpha-SQL、DeepEye-SQL 等 test-time framework 相比，在准确率、预处理成本和在线查询成本之间处于什么位置？

（3）Pontis 的图谱元数据、消歧实体、SQL 验证查询和 BIRD global 经验分别贡献了什么？

（4）即席查询性能主要受哪些因素影响，是最终输出长度、工具调用次数，还是串行 LLM 轮次和输入上下文膨胀？

本文采用 BIRD 开发集作为主要实验数据。BIRD 包含真实数据库、自然语言问题、evidence、标准 SQL 和难度标签，适合评估真实数据库问答能力。当前阶段的 Pontis 日志覆盖 1533 道 dev 题目，后续完整评测将覆盖官方 dev 的 1534 道题。

\subsection{错误分析}

从当前日志看，Pontis 的错误并非单纯来自模型不会写 SQL，而是集中在以下几类：

（1）BIRD SQL 风格偏好。BIRD 中一些题目对百分比、比例、输出列、排序和 \texttt{COUNT DISTINCT} 有较强 benchmark 风格。模型有时会生成业务上合理但与 gold SQL 口径不同的查询。

（2）局部 schema 理解错误。当表名、列名或业务术语相近时，模型可能选择错误字段。例如同名 \texttt{name}、\texttt{id}、\texttt{type} 字段需要结合表语义和样例值消歧。

（3）value grounding 错误。模型需要把自然语言中的实体值映射到数据库中的实际字符串、枚举或编码。若样例值检索不足，容易产生条件偏差。

（4）JOIN 路径偏差。复杂数据库中正确连接路径可能需要桥接表或中间关系。即使模型识别了相关表，也可能遗漏桥接表或连接到错误列。

（5）工具行为问题。过度探索、上下文过长、工具结果截断和 SQL 验证不足都可能导致最终输出偏离正确答案。

这些错误类型对应了 Pontis 后续优化方向：加强 BIRD global 经验的约束力、改进局部消歧实体、降低工具轮次、增强 join 路径检查和 value retrieval 能力。

\subsection{性能公平性讨论}

不同方法的性能不能只用外层 \texttt{time} 粗略比较。Bash Agent 几乎没有预处理，但每题都要重新探索；Pontis 有图谱和 embedding 预处理，在线阶段通过工具读取结构化知识；CHESS、Alpha-SQL 和 DeepEye-SQL 使用多阶段 pipeline、候选生成、搜索或 revision，其 LLM 调用数和候选数可能显著高于 agent 方法。

因此，本文采用两层计时协议。第一层用统一脚本记录每个命令的 wall-clock、退出码、环境变量、worker 数和 cache 状态。第二层解析各方法内部日志，提取每题耗时、LLM 调用、token、工具调用和 SQL 执行次数。主性能表只报告冷启动预处理和冷启动在线解题；warm cache 和断点续跑作为系统分析附表。

在预期趋势上，Bash Agent 的预处理成本最低，但准确率和稳定性较弱；Pontis 预处理中等，在线成本取决于 agent loop 是否收敛；CHESS 的 pipeline 成本中高；DeepEye-SQL 的多阶段 snapshot 和 revision 成本较高；Alpha-SQL 的 MCTS rollout 可能带来最高在线 LLM 调用量。Pontis 的目标不是绝对最快，而是在准确率、可解释性、缓存能力和在线成本之间取得较好平衡。

\section{总结与展望}

\subsection{全文总结}

本文针对真实数据项目中的自然语言分析问题，设计并实现了 Pontis 图谱化智能体工作空间。Pontis 的核心思想是在原始数据和 LLM 智能体之间建立可查询、可更新、可复用的知识图谱层，使智能体能够通过工具逐步获取精确上下文，而不是在每道题中从零读取全部 schema 和数据。

本文完成了以下工作：

（1）设计了以 Neo4j 和 Cypher 为核心的图谱工作空间，将项目中的文件、数据库、表、列、外键、统计摘要、语义描述和经验规则统一建模。

（2）实现了自动化脚本和智能体探索结合的图谱构建机制，使系统能够按需发布文件系统和 SQLite schema 等源事实，并自动生成列统计、值分布、列重叠、AI 摘要、语义向量和消歧知识等派生知识。

（3）实现了面向数据分析的工具调用智能体，支持图检索、元数据查看、SQL 执行、文本搜索、JSON 查看和图写入等能力。

（4）引入了多类护栏机制，对 SQL 实体、桥接表、同名列消歧、过度探索和工具轮次进行约束，减少 LLM 在数据库问答中的常见错误。

（5）在 BIRD dev 上进行了阶段性评估，并提出了与 Bash Agent、CHESS、Alpha-SQL、DeepEye-SQL 等方法进行公平比较的完整实验协议。

\subsection{不足与展望}

当前系统仍存在若干不足。第一，Pontis 的在线表现仍受 agent loop 轮次影响，复杂问题可能需要较多工具调用，导致每题耗时达到分钟级。第二，BIRD global 经验虽然可被检索到，但模型并不总能将其作为最终 SQL 的硬约束。第三，现阶段对增量更新成本的实验还不完整，尚未系统验证图谱实体级更新相比 pipeline rebuild 的优势。第四，baseline 的完整复现实验仍需补齐，以获得更可靠的横向比较结果。

未来工作可从以下方向展开：

（1）优化工具调度策略，减少不必要的 \texttt{find}、\texttt{meta} 和 \texttt{query} 轮次。

（2）增强经验库约束机制，使抽象规则能更稳定地影响最终 SQL。

（3）系统评估增量更新能力，例如新增列、修改值分布、新增关系和删除表列后的局部刷新成本。

（4）扩展数据源类型，支持远程数据库、对象存储、日志系统和业务 API。

（5）引入更细粒度的性能日志，持续监控 LLM round、input token、tool result token 和 SQL execution 对在线延迟的影响。

总体而言，Pontis 展示了一条区别于专用 SQL 模型和重型候选搜索 pipeline 的路线：通过维护长期存在的数据项目工作空间，让通用 LLM 智能体在真实数据环境中获得可解释、可复用、可演化的结构化记忆。
